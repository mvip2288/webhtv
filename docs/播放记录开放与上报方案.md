# 观影记录同步方案

状态：分阶段实现；Webhook 上报和当前播放只读 API 已实现，本机写入 API、远端同步源和新 UI 待实现
日期：2026-06-12

## 1. 目标与入口

本方案的产品名称统一为 **观影记录同步**。它覆盖三类能力：

1. **本机 API 读写**：JS/Python/PHP/Jar 等爬虫调用 App 本机 HTTP API，读取当前播放记录，或按字段规范把远端观影记录写回 App 本地历史。
2. **远端同步**：用户在 App 中配置远端 API，App 主动从该 API 拉取观影记录并合并到本地 `History`。
3. **Webhook 上报**：用户在 App 中配置端点，App 在播放过程中主动 POST 观影进度到用户服务器。

不提供远程静默配置入口。远程点播配置、站点配置、CSP、`extensions`、`webHomeExtensions` 都不能静默下发观影记录同步配置；WebHome 既有 `fm.history()`/`app.history` 属于原有历史能力，不是本方案新增入口。WebHome 可以像普通网页一样调用本机 HTTP API，但它不是唯一同步通道；更通用的方式是由 JS/Python/PHP/Jar 等爬虫或 App 内置远端同步能力调用统一写入逻辑。

### 1.1 UI 信息架构

增强功能入口名称改为 **观影记录同步**。顶层界面只做总览和分流，不直接展示长列表：

1. **开关**
   - 观影记录同步总开关。
   - 本机 API 写入开关：控制 `/api/playback/progress` 和 `/api/playback/progress/batch` 是否允许外部写入。
   - 本机 API token：可选；用于本机写入 API 鉴权。关闭写入开关时，爬虫仍可读取当前播放记录，但不能写入本地历史。
2. **远端同步**
   - 顶层显示启用数量、最近同步时间、最近结果、最近错误。
   - 点击后进入远端同步源列表。列表只显示核心摘要和常用操作；新增/修改在独立弹窗中完成。
3. **Webhook 上报**
   - 顶层显示启用数量、最近上报状态、最近错误。
   - 点击后进入 Webhook 端点列表。沿用当前摘要列表 + 独立新增/修改弹窗模式。
   - 不提供“上报时机”选择。Webhook 固定使用最佳实践事件：周期进度、暂停/退出/切集前最终进度、自然完播。

目标结构：

```text
观影记录同步
├─ 开关
│  ├─ 总开关
│  ├─ 允许本机 API 写入
│  └─ 本机 API token
├─ 远端同步
│  └─ 远端同步源列表
│     └─ 新增/编辑远端同步源
└─ Webhook 上报
   └─ Webhook 端点列表
      └─ 新增/编辑 Webhook
```

设计原则：

- 顶层只展示状态摘要，不放长表单。
- “远端同步”和“Webhook 上报”都是二级功能，点击后才进入各自列表。
- 列表只显示核心摘要、启用状态、最近状态和常用操作。
- 新增/修改全部使用独立弹窗，不在列表行内展开复杂表单。
- 本机 API 写入开关和远端同步不强绑定。远端同步是 App 内部能力，关闭本机 API 写入后仍可由 App 自己拉取并合并远端记录。

## 3. 爬虫 HTTP API 文档

爬虫侧使用本机 HTTP REST API 获取当前播放记录，适用于 JS、Python、PHP、Jar 等不同运行时。Java `PlaybackApi.current(siteKey)` 仅作为 App 内部复用 helper，不作为对外接入方式。

API 只读当前播放项，不提供全局历史列表。调用时必须传 `siteKey`，App 会校验该 `siteKey` 与当前播放记录一致，避免一个站点读取另一个站点的当前播放数据。

### 3.1 Endpoint

```http
GET http://127.0.0.1:{port}/api/playback/current?siteKey={siteKey}
```

兼容短路径：

```http
GET http://127.0.0.1:{port}/playback/current?siteKey={siteKey}
```

说明：

- `{port}` 为 App 本机 HTTP 服务端口。JS Spider 可用 `getPort()` 获取；Jar 爬虫可用 `com.github.catvod.Proxy.getPort()` 获取；其它运行时只要能访问 App 本机或局域网 HTTP 地址即可。
- 也可以使用局域网地址：`http://设备IP:{port}/api/playback/current?siteKey={siteKey}`。
- 接口支持 `GET` 和 `POST`，推荐用 `GET`。
- 返回内容是播放记录 JSON 本体，不包 `code/data` 外壳，方便不同语言直接解析。

### 3.2 参数

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `siteKey` | string | 是 | 调用方站点 key。必须与当前播放记录的 `siteKey` 完全一致 |
| `site` | string | 否 | `siteKey` 的兼容别名；同时存在时优先使用 `siteKey` |

### 3.3 HTTP 状态码

| 状态码 | 含义 | Body |
| --- | --- | --- |
| `200` | 成功读取当前播放记录 | 播放记录 JSON 本体 |
| `400` | 缺少 `siteKey` | `{"code":400,"message":"siteKey不能为空"}` |
| `404` | 当前无可读播放记录，或当前播放项不属于该站点 | `{"code":404,"message":"当前无可读播放记录"}` |
| `405` | 方法不支持 | `{"code":405,"message":"只支持 GET/POST"}` |
| `500` | 内部错误 | `{"code":500,"message":"..."}` |

### 3.4 成功响应示例

```json
{
  "schema": "webhtv.playback.v1",
  "timestamp": 1781170000000,
  "sessionId": "2a9d6c9f-...",

  "historyKey": "siteKey@@@vodId",
  "siteKey": "siteKey",

  "vodPic": "https://example.com/poster.jpg",
  "flag": "线路1",

  "positionMs": 123456,
  "durationMs": 456789,
  "progress": 0.2702,
  "speed": 1.0,
  "completed": false
}
```

### 3.6 返回空的情况

- `siteKey` 为空。
- 当前是隐身模式。
- 当前没有有效播放记录。
- 当前播放项不属于调用方站点。
- 当前播放记录还没有可保存的 `positionMs` / `durationMs`。

### 3.7 API 字段范围

爬虫 HTTP API 返回 `apiSafe` 字段集：`schema/timestamp/sessionId/dedupeKey/cid/historyKey/siteKey/siteName/vodId/vodName/vodPic/flag/episodeName/state/positionMs/durationMs/progress/speed/completed`。

API 不返回 `event`、`eventId`、`episodeUrl`、真实播放地址、headers、cookies、配置地址、Jar/ext、设备标识、全局历史或其他站点历史。

## 4. App Webhook 配置

Webhook 是用户本地配置，配置项包括：

| 配置项 | 默认值 | 说明 |
| --- | --- | --- |
| `name` | 空 | 可选；为空时 UI 使用端点 URL 的 host 作为名称，不包含协议和 path |
| `enabled` | true | 启用/停用端点 |
| `url` | 空 | 必填，必须以 `http` 开头 |
| `token` | 空 | 可选；仅作为 `X-WebHTV-Token` 请求头发送，不参与 `dedupeKey` 计算 |
| `siteKeys` | 空 | 可选；为空表示全部站点 |
| `fieldPreset` | `basic` | 字段预设；UI 显示为“基础/标准/完整/自定义” |
| `fields` | 空 | 仅自定义预设使用，由 UI 多选，不使用手填 key；每个字段必须显示简短说明 |
| `progressIntervalSec` | 30 | progress 节流间隔，0 表示关闭周期进度事件 |
| `maxRetries` | 2 | 失败重试次数，范围 0-3 |

连续失败达到阈值后端点会自动暂停。用户再次点击“启用”或保存启用状态时，会清空暂停和最近错误。

旧配置里的 `events` 字段只作为兼容读取。目标 UI 不展示、不编辑该字段；保存时统一写入固定事件策略。

## 5. 统一上报时机

Webhook 面向普通用户和用户自有服务，不应要求用户理解 `start/stop/pause/resume/ended` 的差异。上报时机统一为一套最佳实践：

| 事件 | 触发说明 |
| --- | --- |
| `playback.progress` | 周期进度事件，默认 30 秒节流 |
| `playback.progress` | 暂停、退出、切集、切线路、播放页销毁前立即补发最终进度 |
| `playback.ended` | 播放器自然完播时发送 |

不发送独立 `playback.start`，服务端可以把某个 `dedupeKey/sessionId` 的第一条 `progress` 视为开始。不发送独立 `pause/resume`，暂停与恢复只影响最终进度和下一次周期进度。不发送独立 `stop`，主动退出用最终 `progress` 表达；自然完播用 `ended` 表达。

底层仍可保留 `start/stop/pause/resume` 常量用于日志、兼容旧数据或后续扩展，但不进入用户配置界面。

## 6. 字段预设与字段说明

| 预设 | 字段 | 说明 |
| --- | --- | --- |
| 基础 | 协议字段 + 播放对象基础信息 + 播放进度 | 默认推荐，适合多数同步服务 |
| 标准 | 基础 + App 版本 + 客户端类型 | 便于服务端排查不同 App 版本和客户端 |
| 完整 | 标准 + 当前集 URL + 当前集序号 + 客户端标识 | 信息最完整，适合用户完全信任的自有服务；当前集 URL 可能包含源站参数 |
| 自定义 | 协议字段固定携带，其他字段由用户多选 | UI 显示字段名和一句说明，不允许手填 key |

协议字段包括：`schema/event/eventId/timestamp/sessionId/dedupeKey`。

当前实现不开放真实播放地址、headers、cookies、配置地址、Jar/ext、本地路径、IP/MAC 等 L4 高风险字段。

内部兼容映射：

| UI 预设 | 内部值 | 兼容说明 |
| --- | --- | --- |
| 基础 | `basic` | 旧 `safe` 配置读取时映射为基础 |
| 标准 | `standard` | 继续兼容旧 `standard` |
| 完整 | `full` | 新增；比标准多 `episodeUrl/episodeIndex/clientKey` |
| 自定义 | `custom` | 继续兼容旧 `custom` |
| 匿名 | 无 UI 入口 | 旧 `anonymous` 配置读取时保留；新增 UI 不再展示匿名预设，避免预设过多 |

| 字段 | 类型 | 出现条件 | 说明 |

| `speed` | number | API、所有 Webhook 预设 | 当前播放倍速 |

## 7. Webhook 接收 API 文档

Webhook 是 App 主动调用用户配置的 HTTP 接口。用户服务器只需要实现一个能接收 JSON 的 `POST` 端点。

### 7.1 Endpoint

```http
POST <用户在 App 中填写的端点 URL>
Content-Type: application/json
```

要求：

- URL 必须以 `http://` 或 `https://` 开头。
- App 不要求固定 path，用户可自行使用 `/api/webhtv/playback`、`/webhook/playback` 等路径。
- Token 可选；如果填写，App 只通过 `X-WebHTV-Token` 请求头发送，不参与 payload 计算。
- App 认为任意 `2xx` HTTP 状态码为成功；非 `2xx`、网络错误或超时都按失败处理并进入重试。

### 7.2 请求头

```http
Content-Type: application/json
X-WebHTV-Timestamp: 1781170000
X-WebHTV-Token: <用户可选 token>
X-WebHTV-Webhook-Id: <eventId>
X-WebHTV-Dedupe-Key: <dedupeKey>
Idempotency-Key: <eventId>
```

| 请求头 | 必填 | 说明 |
| --- | --- | --- |
| `Content-Type` | 是 | 固定为 `application/json` |
| `X-WebHTV-Timestamp` | 是 | 发送请求时的秒级 Unix 时间戳 |
| `X-WebHTV-Token` | 否 | 用户在 App 配置的 token。服务端可用它做用户空间、鉴权或分组 |
| `X-WebHTV-Webhook-Id` | 是 | 与 payload 的 `eventId` 一致 |
| `X-WebHTV-Dedupe-Key` | 是 | 与 payload 的 `dedupeKey` 一致 |
| `Idempotency-Key` | 是 | 与 `eventId` 一致，用于兼容常见 Webhook 幂等处理 |

`token` 只是用户服务器分组/鉴权用的共享值，App 不用它计算任何字段。同一个 token 下，服务端可以用 `dedupeKey` 合并多端同一播放条目，用 `eventId`/`Idempotency-Key` 去重同一次事件重试。

### 7.3 事件名

| 事件 | 是否对外发送 | 触发说明 |
| --- | --- | --- |
| `playback.progress` | 是 | 周期进度事件；默认 30 秒节流。暂停、退出、切集、切线路、完播前会补发最终进度 |
| `playback.ended` | 是 | 播放器自然完播时发送 |
| `playback.start` | 否 | 服务端可用第一条 `progress` 推断开始 |
| `playback.stop` | 否 | 主动退出用最终 `progress` 表达 |
| `playback.pause` / `playback.resume` | 否 | 暂停/恢复不单独对外上报，避免噪声 |

目标 UI 不提供上报时机选择。服务端只需要处理 `progress` 和 `ended`，协议稳定且更易接入。

### 7.5 推荐服务端处理

服务端建议按下面顺序处理：

1. 校验 `Content-Type` 和 JSON body。
2. 如果服务端要求鉴权，校验 `X-WebHTV-Token` 是否属于当前用户。
3. 用 `eventId` 或 `Idempotency-Key` 做事件级幂等，重复事件直接返回 `2xx`。
4. 用 `dedupeKey` 在用户空间内合并同一播放条目。
5. 按 `event` 更新状态：`progress` 更新进度，`ended` 标记完播。

最小 Node.js/Express 示例：

```js
app.post('/api/webhtv/playback', express.json(), async (req, res) => {
  const token = req.header('X-WebHTV-Token') || '';
  const idempotencyKey = req.header('Idempotency-Key') || req.body.eventId;
  const dedupeKey = req.body.dedupeKey;

  if (!dedupeKey || !idempotencyKey) return res.sendStatus(400);
  if (await seenEvent(idempotencyKey)) return res.sendStatus(204);

  await markEventSeen(idempotencyKey);
  await upsertPlayback({ token, dedupeKey, payload: req.body });
  res.sendStatus(204);
});
```

## 8. 去重与身份

- `eventId`：每次事件生成一个 UUID，用于投递幂等和重试去重。
- `dedupeKey`：由播放对象稳定字段生成的 SHA256，用于同一播放条目的合并，不包含 token。
- `clientKey`：本机 Android ID 的 SHA256，仅在“完整/自定义”字段策略中可发送，默认“基础”不发送。

服务端推荐策略：

1. 先按 `X-WebHTV-Token` 或 endpoint 自身归属确定用户空间。
2. 再按 `dedupeKey` 合并播放条目。
3. 再按 `eventId`/`Idempotency-Key` 去重投递。

## 9. 发送策略

- 发送在后台线程执行，不阻塞播放和历史保存。
- `progress` 默认 30 秒节流。
- 暂停、退出、完播时会取消待发送的旧 progress，并在需要时立即补发最终 progress。
- 单个 endpoint 串行发送，避免乱序；不同 endpoint 可并发。
- HTTP 超时为 5 秒。
- 失败最多重试 0-3 次，指数退避。
- 连续失败达到阈值后暂停端点，避免持续打扰服务器或耗电。

## 10. 隐私声明

首次打开配置时显示隐私声明，说明 Webhook 上报会把影片名称、进度等观看记录发送到用户配置的服务器。App 不会收集或转发给第三方；用户可随时停用或删除端点。

## 11. 当前结论

- 功能名称统一为“观影记录同步”。
- 顶层界面分为“开关”“远端同步”“Webhook 上报”三块。
- 本机 API 写入能力由独立开关控制，默认关闭；只读 `/api/playback/current` 不受写入开关影响。
- 远端同步是 App 主动从用户填写的 API 拉取观影记录，并复用同一套写入合并逻辑。
- Webhook 上报继续保留，作为 App 主动把本地播放进度推给用户服务器的能力。
- 不存在 CSP 内联配置静默下发观影记录同步配置。
- Token 可选，只随请求头或本机写入 API 鉴权传递，不参与 `dedupeKey` 计算。
- Webhook 上报时机统一固定为：周期进度、退出/暂停/切集前最终进度、自然完播。
- 默认字段预设为“基础”，可选“标准/完整/自定义”。
- 自定义字段通过 UI 多选，不需要用户手填 key；每个字段必须显示简短说明。

## 12. 观影记录同步方案评审（待实现）

### 12.1 问题定义

现有能力只能把播放记录从 App 发出去，或让爬虫读取当前播放项。它可以完成“上报”，但不能完成“多端恢复进度”。目标态需要三条互补通道：

1. **本机写入 API**：爬虫、WebHome 返回后补偿逻辑或局域网工具把远端观影记录写入 App 本地历史。
2. **远端同步源**：App 主动从用户填写的 API 拉取观影记录，合并到本地历史。
3. **Webhook 上报**：App 在播放过程中把本地进度推送到用户服务器。

三条通道共用同一套 `PlaybackProgressInput -> PlaybackProgressWriter -> History` 合并逻辑，避免远端同步、本机写入和后续内部调用出现不同冲突规则。

### 12.2 现有代码评估

当前播放进度最终落在 `History` 表：

| `History` 字段 | 含义 | 对应开放字段 |
| --- | --- | --- |
| `key` | 历史主键，当前由 `siteKey + AppDatabase.SYMBOL + vodId` 组成 | `historyKey` 或 `siteKey + vodId` |
| `vodName` | 影片名 | `vodName` |
| `vodPic` | 封面 | `vodPic` |
| `vodFlag` | 线路/播放源 | `flag` |
| `vodRemarks` | 当前集名称 | `episodeName` |
| `episodeUrl` | 当前集 URL | `episodeUrl` |
| `position` | 播放进度，毫秒 | `positionMs` |
| `duration` | 视频总时长，毫秒 | `durationMs` |
| `speed` | 倍速 | `speed` |
| `createTime` | 最近进度时间 | `updatedAt` |
| `cid` | 当前点播配置 id | 当前运行环境 `VodConfig.getCid()` |

可复用点：

- `VodBrowse.saveProgress(...)` 已经证明 `History.position/duration/createTime` 是恢复进度所需的核心状态。
- `History.sync(...)` 已经存在批量合并历史的能力。
- `Action.syncHistory(...)` 已经有局域网设备间批量同步历史的路径。
- 当前 `PlaybackWebhookDialog` 已经采用“主列表摘要 + 新增/编辑独立弹窗”的交互方式，可下沉为二级 Webhook 列表。
- `HomeWebBridge` 的 `playUrl/playVod/playVodInline/playPan` 都会调用 `VideoActivity.start(...)` 进入原生播放页；`HomeWebController` 只在 WebView 恢复时派发 `fmresume`；`fm.history()` 读取的是当前点播配置下的 `History.get()`。

不足：

- `History.sync(...)` 当前按 `vodName` 找同名记录，再用 `createTime` 判断是否覆盖；这适合同步整份 App 历史，但不够适合开放给爬虫和远端同步源的精确进度写入。
- `History.updateTime` 是 transient，只用于进程内节流，不落库，不能作为跨设备冲突依据。
- 当前只读 `PlaybackRecord` 不输出 `episodeUrl`，但写入时应允许调用方提供 `episodeUrl` 作为同集判定辅助字段。
- 当前 API 只读 `/api/playback/current`，没有单条写入和批量写入 endpoint。
- 当前配置入口名称和 UI 只覆盖 Webhook 上报，不覆盖远端拉取和本机写入开关。
- WebHome 发起原生播放后，播放期间不能可靠依赖 WebView JS 继续运行或主动轮询 SDK/API。WebHome 只能在返回后通过 `fmresume + fm.history()` 或本机 API 做补偿；实时同步必须由 App 播放服务、Webhook、本机写入和远端同步承担。

结论：底层可行，但需要把观影进度写入抽成独立领域能力。不能直接开放完整 `History` JSON，也不能把远端同步做成 Webhook 配置里的高级字段。

### 12.2.1 WebHome 同步边界

WebHome 不是播放期间的实时同步执行者。原因是 WebHome 调起原生播放后会进入 `VideoActivity`，当前 WebView 处于后台/不可交互状态，JS 定时器、SDK 调用和本机 HTTP 轮询都不应作为可靠链路。

推荐分工：

| 场景 | 推荐同步方式 | 说明 |
| --- | --- | --- |
| 播放期间实时上报 | App 级 Webhook 上报 | 由 `PlaybackEventCollector` 从原生播放服务收集进度，不依赖 WebView |
| 多端拉取远端进度 | App 级远端同步源 | App 主动拉取用户 API 并合并到 `History` |
| WebHome 返回后补偿 | `fmresume` 后调用 `fm.history()` 或 `/api/playback/current` | 适合 Nostr 首页这类页面刷新本地状态、补发页面自己的事件 |
| WebHome 写入远端进度到 App | 返回后调用本机写入 API | 只适合 WebHome 可见时补偿，不适合播放期间实时写入 |

WebHome 发起播放时也要注意身份稳定性：

1. `fm.vod(siteKey, vodId, ...)` 最适合观影记录同步，App 历史主键保留原站点 `siteKey + vodId`。
2. `playVodInline` 会使用 WebHome inline 虚拟站点，能保存历史，但多端合并依赖稳定的 inline `vodId`、集名和集 URL。
3. `playUrl`、`pan.play`、push 直链通常只按 `PUSH + url/title` 形成历史，跨端去重和合并准确性最低。

### 12.4 本机写入 API

新增两个本机 HTTP endpoint：

```http
POST /api/playback/progress
POST /api/playback/progress/batch
```

兼容短路径：

```http
POST /playback/progress
POST /playback/progress/batch
```

说明：

- 只支持 `POST` 和 `OPTIONS`。
- `Content-Type` 推荐 `application/json`。
- 支持 CORS，与 `/api/playback/current` 保持一致。
- 写入能力由“允许本机 API 写入”开关控制，默认关闭。
- 如果用户配置了本机 API token，调用方通过 `Authorization: Bearer <token>`、`X-WebHTV-Token: <token>` 或 query `token=<token>` 传入；三者任选其一，推荐 `Authorization`。
- 关闭写入开关时，爬虫仍可调用 `/api/playback/current` 读取当前播放记录，但不能写入本地历史。

### 12.5 远端同步源 API

远端同步源是用户在 App 中配置的 HTTP API。App 调用该 API 拉取观影记录，再使用本地 `PlaybackProgressWriter` 合并。

远端同步的用户空间由远端 API 按 token 区分。同一个远端 URL 下，A/B/C 三个 App 客户端配置同一个 token 时，应拉取同一套远端观影记录；D/E 配置另一个 token 时，应拉取另一套记录。App 只负责把 token 随请求发给远端，不把 token 写入本地历史主键，也不把 token 纳入 `dedupeKey` 计算。

推荐远端 API：

```http
GET <用户填写的远端 API>?since=<毫秒时间戳>&limit=100
Authorization: Bearer <用户填写的 token>
```

远端也可以只支持固定 URL，不处理 query。App 仍应能读取返回的 `items`。

远端响应格式：

```json
{
  "items": [
    {
      "siteKey": "site_key",
      "vodId": "vod_id",
      "vodName": "影片名",
      "vodPic": "https://example.com/poster.jpg",
      "flag": "线路1",
      "episodeName": "第1集",
      "episodeUrl": "https://example.com/play/1.m3u8",
      "positionMs": 123456,
      "durationMs": 456789,
      "speed": 1.0,
      "completed": false,
      "updatedAt": 1781170000000
    }
  ],
  "nextSince": 1781170000000
}
```

远端同步源配置字段：

| 配置项 | 默认值 | 说明 |
| --- | --- | --- |
| `name` | 空 | 可选；为空时 UI 使用 API URL 的 host |
| `enabled` | true | 启用/停用该同步源 |
| `url` | 空 | 必填，必须以 `http` 开头 |
| `token` | 空 | 可选；随请求头发送，由远端用于鉴权和用户数据分组，不参与本地合并主键 |
| `siteKeys` | 空 | 可选；为空表示接受全部站点 |
| `syncOnStartup` | true | App 启动或进入增强页后可自动同步 |
| `syncBeforePlayback` | false | 进入播放前尝试同步当前站点相关记录；默认关闭，避免阻塞播放 |
| `intervalMinutes` | 0 | 定时同步间隔；0 表示不定时 |
| `limit` | 100 | 单次最大拉取条数 |
| `timeoutSec` | 5 | HTTP 超时 |
| `lastSyncAt` | 0 | 最近成功同步时间 |
| `lastError` | 空 | 最近错误 |

二级“远端同步”列表显示：

- 名称。
- 启用状态。
- API host/path 摘要。
- 最近同步时间。
- 最近同步结果：拉取条数、写入条数、跳过条数、失败条数。
- 最近错误。
- 操作：立即同步、修改、删除。

新增/编辑远端同步源使用独立弹窗。高级设置默认折叠。


批量规则：

- `items` 最大建议 100 条，避免一次请求长时间占用数据库。
- 单条失败不影响其它条目。
- 响应逐条返回状态，方便调用方重试失败项。
- 批量写入和远端同步合并应在后台线程执行，完成后触发历史列表刷新事件。

### 12.8 写入响应格式

单条成功：

```json
{
  "code": 0,
  "status": "updated",
  "key": "site_key@@@vod_id",
  "message": "ok"
}
```

批量响应：

```json
{
  "code": 0,
  "updated": 2,
  "skipped": 1,
  "failed": 1,
  "items": [
    { "index": 0, "key": "site@@@vod1", "status": "updated" },
    { "index": 1, "key": "site@@@vod2", "status": "skipped", "reason": "older" },
    { "index": 2, "status": "failed", "reason": "duration invalid" }
  ]
}
```

HTTP 状态码建议：

| 状态码 | 含义 |
| --- | --- |
| `200` | 请求已处理；批量中可能有部分失败，详见 body |
| `400` | JSON 无效、缺少 `items`、字段类型错误 |
| `401` | token 缺失或错误 |
| `403` | 写入开关关闭、隐身模式、站点不允许写入 |
| `405` | 方法不支持 |
| `413` | 批量条数过多或 body 过大 |
| `500` | 内部错误 |

### 12.9 校验与合并策略

写入前校验：

1. 隐身模式开启时拒绝写入。
2. 本机 API 写入请求在写入开关关闭时拒绝；App 内部远端同步不受该开关限制，但受总开关和远端源启用状态限制。
3. token 配置不为空时必须匹配。
4. `siteKey/vodId/vodName/episodeName` 不能为空。
5. `positionMs >= 0`。
6. `durationMs > 0`。
7. `positionMs <= durationMs`；允许少量误差时可 clamp 到 `durationMs`。
8. `updatedAt` 必须为合理时间戳；过旧或明显未来的时间应拒绝或降级处理。

合并策略：

1. 主定位：`siteKey + vodId`。
2. 剧集定位：优先比较 `episodeUrl`，其次比较 `flag + episodeName`，最后只比较 `episodeName`。
3. 如果本地没有记录，创建新 `History`。
4. 如果本地有同一影片但不同集，写入远端集信息和进度。
5. 如果本地同一影片同一集存在记录，远端 `updatedAt > local.createTime` 时覆盖进度。
6. 如果没有 `updatedAt`，只允许更高 `positionMs` 覆盖，且不得覆盖本地已接近完播的记录。
7. `completed=true` 或 `positionMs/durationMs >= 0.95` 可视为完播；本地仍保存进度为 `durationMs` 或接近片尾的位置，避免下次从 0 开始。
8. 当前正在播放同一条目时，不强制 seek；只写数据库。否则外部同步可能打断用户当前播放。

和现有 `History.sync(...)` 的差异：

- 不按 `vodName` 作为唯一合并依据，避免同名影片误合并。
- 不直接接收完整 `History` JSON，避免外部写入 `opening/ending/revSort/revPlay/scale/cid` 等内部字段。
- 不要求调用方知道 `AppDatabase.SYMBOL`，但兼容 `historyKey`。
- 使用 `updatedAt` 明确表达远端进度新旧，而不是复用事件投递时间。

### 12.10 与 Webhook 上报的字段关系

Webhook 上报字段、远端同步字段和本机写入字段应保持语义一致，但不是完全同一集合：

| Webhook/API 读取字段 | 写入/远端同步字段 | 说明 |
| --- | --- | --- |
| `siteKey` | `siteKey` | 一致 |
| `vodId` | `vodId` | 一致 |
| `vodName` | `vodName` | 一致 |
| `vodPic` | `vodPic` | 一致 |
| `flag` | `flag` | 一致 |
| `episodeName` | `episodeName` | 一致 |
| `episodeUrl` | `episodeUrl` | 读取默认不返回，写入和远端同步建议支持 |
| `positionMs` | `positionMs` | 一致 |
| `durationMs` | `durationMs` | 一致 |
| `speed` | `speed` | 一致 |
| `completed` | `completed` | 写入时作为辅助，最终以进度和阈值校验 |
| `timestamp` | `updatedAt` | 读取/上报是事件生成时间；写入应使用远端记录更新时间 |
| `dedupeKey` | 不作为必填 | 可用于服务端去重，但 App 本地写入仍以 `siteKey/vodId/episode` 合并 |
| `sessionId/eventId` | 不写入 | 事件级字段，不属于本地历史状态 |

### 12.11 实现建议

新增组件：

| 组件 | 职责 |
| --- | --- |
| `PlaybackProgressInput` | 写入 DTO，解析单条 JSON |
| `PlaybackProgressApplyResult` | 单条应用结果 |
| `PlaybackProgressWriter` | 校验、合并、写入 `History`；本机写入 API 和远端同步共用 |
| `PlaybackProgressApi` | HTTP endpoint：`/api/playback/progress`、`/api/playback/progress/batch` |
| `PlaybackRemoteSyncStore` / `RemoteSyncConfig` | 远端同步源配置、最近状态、最近错误 |
| `PlaybackRemoteSyncWorker` | 从远端 API 拉取 `items` 并调用 `PlaybackProgressWriter` |
| `ViewingRecordSyncDialog` | 顶层“观影记录同步”界面 |
| `PlaybackRemoteSyncDialog` | 二级“远端同步”列表和新增/编辑弹窗 |
| `PlaybackWebhookDialog` | 下沉为二级“Webhook 上报”列表 |

优先级：

1. 实现“观影记录同步”顶层入口和 UI，顶层分三块。
2. 实现 `PlaybackProgressInput` 和 `PlaybackProgressWriter`。
3. 实现本机单条/批量写入 API 和写入开关。
4. 实现远端同步源配置、列表、立即同步。
5. 增加远端同步的启动时同步、间隔同步和最近状态。
6. 更新 README、完整开发文档和本方案。
7. 增加基础测试：字段校验、旧记录跳过、新记录写入、批量部分失败、隐身模式拒绝、token 校验、远端同步源失败状态。

暂不做：

- App 内置云端账号体系。
- 播放中收到远端进度后自动 seek。
- 暴露完整 `History` 增删改查。
- 让远程点播配置或 CSP 静默下发同步端点。

### 12.12 最终 review 结论

该方案可行，且 UI 设计方向正确。原因：

1. 真实目标是多端观影记录同步，仅上报和查询不能形成闭环。
2. 当前代码已有 `History` 写入和批量同步基础，不需要重做存储层。
3. 外部成熟系统普遍把播放进度作为独立 playstate/progress API 处理，而不是只依赖 watched 状态。
4. “观影记录同步”顶层三块结构能把本机写入、远端拉取、Webhook 推送三个概念分开，避免在一个列表里堆过多字段。
5. 远端同步源列表和 Webhook 端点列表都采用“摘要列表 + 独立新增/编辑弹窗”，与当前 Webhook UI 和最近的扩展/站点注入改造风格一致。
6. API 必须字段受控、支持批量、支持 token、支持冲突策略，不能直接开放完整 `History` JSON。
